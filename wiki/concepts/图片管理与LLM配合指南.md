# 图片管理与 LLM 配合指南

> 本文档总结在 Obsidian 知识库中管理图片的最佳实践，特别是面向亚马逊美工部门的高图片密度场景，以及如何与 LLM 高效配合。

## 1. Obsidian 图片能力一览

| 能力 | 说明 |
|------|------|
| **原生嵌入** | `! [ [ 图片.png ] ]`（实际使用时去掉空格）直接渲染显示，与图谱视图联动 |
| **附件管理** | 可指定统一附件目录（如 `raw/assets/`），所有图片自动归集 |
| **一键下载** | Obsidian Web Clipper 抓取网页后，Ctrl+Shift+D 将远程图片拉到本地 |
| **多图对比** | 结合 Image Layouts 等插件可并排对比多张图片 |

Obsidian 展示图片没有问题。真正的关键在于 **LLM 如何读取图片**。

---

## 2. LLM 读取图片的方式

LLM（如 Claude）是**多模态**的——可以直接看懂图片内容（构图、配色、文字可读性、产品展示角度等）。

但由于工作方式限制，最佳配合策略如下：

### 2.1 图文分离，文本优先

每张重要图片前写一段文字说明，LLM 优先读取文本获取上下文：

```markdown
# 产品A - A+设计稿

## 设计说明
- 主色调：深蓝 #003366
- 排版风格：左文右图
- 重点突出：产品防水功能

## 参考图片
! [ [ raw/assets/product-a-aplus-main.jpg ] ]
```

**效果**：LLM 先理解上下文，再按需看图，效率和准确性更高。

### 2.2 给图片写说明（Alt Text）

```markdown
! [ [ 主图规范-示例.jpg ] ]
*↑ 示例：白底主图，产品居中，占画面约 80%，右侧 45° 角展示，下方留 10% 空白放 logo*
```

这样即使 LLM 不立即看图，也能通过描述理解内容；需要看细节时它会主动读取图片。

### 2.3 主动要求 LLM 审图

在提问时直接附上图片，LLM 可以进行分析：

> "帮我看看这张 A+ 设计图有没有问题 → `! [ [ design-v3.jpg ] ]`"
> "对比这两张主图 → `! [ [ main-v1.jpg ] ]` 和 `! [ [ main-v2.jpg ] ]`，哪个更好？"

LLM 可分析维度：配色协调性、构图、文字可读性、产品展示角度、信息层级等。

## 3. 图片目录结构建议

```
raw/
├── assets/                         ← 所有本地化图片
│   ├── product-photos/             ← 产品原始照片
│   │   └── [ASIN-XXX]/             ← 按 ASIN 分组
│   │       ├── main.jpg
│   │       ├── angle-1.jpg
│   │       └── infographic.jpg
│   ├── templates/                  ← 模板截图
│   ├── competitor-ads/             ← 竞品截图
│   └── references/                 ← 灵感参考
│
wiki/
└── products/
    └── [产品ASIN]/
        ├── 设计稿.md               ← 文字说明 + 图片嵌入
        └── 修改记录.md             ← 版本对比
```

### 命名规范

```
[ASIN]-[模块]-[版本号].jpg
```

示例：`B0ABCDEFG-main-v3.jpg`

好处：可搜索、可排序、一目了然。

## 4. 推荐工具与插件

| 工具/插件 | 用途 |
|-----------|------|
| **Obsidian Web Clipper** | 从竞品页面、设计灵感网站一键剪藏到 `raw/` |
| **Download Attachments**（内置） | 设置热键 Ctrl+Shift+D，批量下载文章内所有图片到本地 |
| **Excalidraw** | 在 Obsidian 内画设计稿标注、流程图、规范示意图 |
| **Image Layouts** | 在笔记中并排对比多张图片（如 A+ 改版前后对比） |
| **git** | Wiki 是 git 仓库，图片也有版本历史——可追溯修改记录 |

---

## 5. 图文配合的典型工作流

当你在资料中放入图片后，LLM 的处理流程：

```
Step 1 → 先读文字部分，理解规范要点和上下文
Step 2 → 逐一查看每张示例图片，理解视觉标准
Step 3 → 在 wiki/ 下创建结构化的页面
Step 4 → 文字中嵌入图片，保持图文关联
Step 5 → 更新索引 (index.md) 和日志 (log.md)
```

---

## 6. 总结

- Obsidian 展示图片：**没有问题**
- LLM 读取图片：**也没有问题**（多模态）
- 关键要点：**图文配合得当**——先文字后图片、重要图片配说明、按 ASIN/项目组织图片目录
