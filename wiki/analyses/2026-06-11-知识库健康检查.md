# 知识库健康检查

> 本次检查范围：`wiki/` 下全部 Markdown 页面、`wiki/index.md`、`wiki/log.md`、`AGENTS.md`，并对照 `raw/` 中已存在资料进行断链和结构检查。

## 总体判断

知识库主结构是可用的，核心链路已经成立：

```text
standards/ 规范
templates/ 模板
cases/ 案例
products/ 产品
workflows/ 工作流
concepts/ 方法论
sources/ 资料摘要
```

主要问题不是内容方向错误，而是近期快速迭代后出现了几类维护债：

- 模式文档 `AGENTS.md` 落后于实际目录
- 同名页面出现两个版本
- 少量“待创建”链接还没有落地
- 个别 Obsidian 目录链接写法不准确
- 示例文档中的占位图片会被自动检查误判为断链

## 发现的问题

### 1. `AGENTS.md` 目录结构落后

实际已经存在：

- `wiki/templates/`
- `wiki/cases/`
- `wiki/products/`
- `wiki/analyses/`

但 `AGENTS.md` 的目录结构目前只写到：

- `wiki/entities/`
- `wiki/concepts/`
- `wiki/standards/`
- `wiki/workflows/`
- `wiki/sources/`

影响：后续 LLM 可能不知道模板、案例、产品页、分析页的正式归属，容易把内容放错目录。

建议：更新 `AGENTS.md`，补充 `templates/`、`cases/`、`products/`、`analyses/`，并在摄入流程中说明这些页面何时更新。

### 2. `AI作图任务标准输入表` 有两个版本

当前存在：

- [[wiki/templates/AI作图任务标准输入表]]：正式模板版本
- `wiki/workflows/AI作图任务标准输入表.md`：早期版本（已删除）

影响：同名页面会造成 Obsidian 搜索和 LLM 引用混乱；用户可能不知道该用哪一个。

建议：

- 保留正式版本：[[wiki/templates/AI作图任务标准输入表]]
- 将早期版本改名为 `AI作图任务标准输入表-旧版.md`，或在页首标记“已废弃，改用正式模板”
- 更新 [[wiki/workflows/AI驱动美工工作流]] 中的引用，统一指向正式模板

### 3. 少量“待创建”链接未落地

发现这些待创建链接：

- [[wiki/concepts/提示词工程]]
- `wiki/concepts/亚马逊图片合规.md`
- [[wiki/workflows/AI生图提示词模板]]

其中：

- `亚马逊图片合规` 的内容已基本被 [[wiki/concepts/AI内容侵权风险与合规]] 覆盖
- `提示词工程` 和 `AI生图提示词模板` 对部门后续使用 GPT 作图有价值，建议创建

建议：

- 把 `wiki/concepts/亚马逊图片合规.md` 的引用改到 [[wiki/concepts/AI内容侵权风险与合规]]
- 创建 [[wiki/concepts/提示词工程]]
- 创建 [[wiki/workflows/AI生图提示词模板]] 或改放到 `wiki/templates/` 下

### 4. 目录链接写法不准确

在 [[wiki/workflows/AI驱动美工工作流]] 的 Mermaid 流程图和部分正文中出现：

- `wiki/cases/` 目录链接写法
- `wiki/products/` 目录链接写法
- `wiki/standards/` 目录链接写法

影响：Obsidian 更稳定的写法是链接到具体索引页，而不是目录。

建议改为：

- `[[wiki/cases/index|案例库]]`
- `[[wiki/products/index|产品知识库]]`
- `[[wiki/index|规范页面]]` 或具体规范页

### 5. 示例占位图片会造成断链误报

[[wiki/concepts/图片管理与LLM配合指南]] 里包含教学示例：

- `! [ [ 图片.png ] ]`
- `! [ [ raw/assets/product-a-aplus-main.jpg ] ]`
- `! [ [ design-v3.jpg ] ]`
- `! [ [ main-v1.jpg ] ]`
- `! [ [ main-v2.jpg ] ]`
- `! [ [ 引用图片 ] ]`

这些是示例，不是真实素材引用。

影响：自动断链检查会误报；用户阅读时也可能以为这些图片应存在。

建议：把这些示例放进代码块，或改成“示例路径：`! [ [ xxx.jpg ] ]`（实际使用时去掉空格）”，避免 Obsidian 当作真实嵌入。

### 6. 日志中有 raw 链接被检测为断链

例如：

- `[[raw/articles/2026-06-09-AI三分钟生成亚马逊主图]]`
- `[[raw/articles/和GPT的沟通]]`
- `[[raw/782-张苏皖新娘派对装饰产品图片文档]]`

这些对应文件实际存在，多数是因为脚本只把 `wiki/` 页面当作可链接页面，没有把 `raw/` 纳入 note link 集合。

判断：这不是实际断链，但后续自动检查脚本应把 `raw/` 也纳入。

### 7. 部分页面是骨架状态

当前这些页面明确是骨架或待填充：

- [[wiki/templates/PSD主图模板]]
- [[wiki/templates/A+模板-基础型]]
- [[wiki/templates/广告素材模板]]
- [[wiki/cases/index]]

影响：结构已经建立，但还不能充分支撑 Codex 生成公司风格的 GPT 提示词。

建议：下一批资料优先摄入“开发给美工的作图模板”和“历史优秀案例”，补齐这些页面。

3. 修正 [[wiki/workflows/AI驱动美工工作流]] 中指向旧输入表和目录链接的引用

## 结论

知识库没有发现严重逻辑错误，也没有发现需要删除的无用主体内容。当前主要是“快速搭建后的小型结构债”：

- 目录规范需要同步
- 同名模板需要统一
- 待创建链接需要落地或改指向
- 示例图片需要避免被当成真实嵌入

按 P1 修完后，知识库就会更适合作为 Codex 调用 GPT 作图时的稳定参考底座。
