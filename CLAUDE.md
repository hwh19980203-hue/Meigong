# 知识库模式文档 (Knowledge Base Schema)

> 本文档定义了 LLM 如何维护本知识库。所有 LLM 会话都应该遵循本文档的约定。

## 架构总览

本知识库采用 [[LLM]] 文档描述的三层架构：

```
raw/             ← 原始资料（不可修改，由用户维护）
wiki/            ← LLM 生成的 Wiki 页面（LLM 完全维护，用户阅读）
CLAUDE.md        ← 模式文档（本文档，用户与 LLM 共同进化）
```

## 目录结构

```
/
├── CLAUDE.md            ← 本文档 — LLM 的工作指令
├── LLM.md               ← 原始设计文档
├── 欢迎.md               ← 欢迎页面
│
├── raw/                 ← 原始资料（用户存放，LLM 只读）
│   ├── _README.md       ← 使用说明
│   ├── assets/          ← 本地化图片/附件
│   ├── articles/        ← 文章
│   ├── books/           ← 书籍
│   ├── podcasts/        ← 播客
│   ├── videos/          ← 视频
│   ├── papers/          ← 论文
│   ├── images/          ← 截图
│   └── sources/         ← 其他外部知识库快照
│
└── wiki/                ← Wiki 知识库（LLM 维护）
    ├── index.md         ← 内容索引（LLM 维护）
    ├── log.md           ← 操作日志（LLM 维护）
    ├── entities/        ← 实体页面（人物、组织、作品等）
    ├── concepts/        ← 概念页面（理论、方法论、模式等）
    ├── standards/       ← 规范标准页面（主图规范、A+规范等）
    ├── workflows/       ← 工作流页面（SOP、操作流程等）

    └── sources/         ← 资料摘要（每份原始资料的摘要页面）
```

## 工作流

### 1. 摄入 (Ingest)

当用户说"处理这份资料"或"帮我读一下 X"时：

1. **读取**原始资料（从 `raw/` 或指定路径）
2. **与用户讨论**：确认重点、澄清疑问、了解上下文
3. **创建摘要页面**：在 `wiki/sources/` 下创建资料摘要
4. **更新实体页面**：在 `wiki/entities/` 中创建或更新相关实体
5. **更新概念页面**：在 `wiki/concepts/` 中创建或更新相关概念
6. **更新规范页面**：如涉及设计标准，在 `wiki/standards/` 中创建或更新规范页面
7. **更新索引**：在 `wiki/index.md` 中添加/更新条目
8. **记录日志**：在 `wiki/log.md` 追加一条记录

**资料摘要页面的格式建议：**
```yaml
---
title: "资料标题"
source: "来源链接或出处"
date: 2026-06-07
type: article | book | podcast | video | paper
tags: [相关标签]
---
```

**单次摄入可能影响的页面数：** 5-15 个页面（取决于资料的丰富程度）。

### 2. 查询 (Query)

当用户提问时：

1. 先读 `wiki/index.md` 定位相关页面
2. 深入阅读相关页面内容
3. 综合回答，**引用来源**（使用 `[[wiki/sources/xxx]]` 或 `[[wiki/entities/xxx]]`）
4. **好的回答本身可以沉淀到 Wiki 中**——如果回答有长期价值，在 `wiki/` 下创建新的分析页面（如 `wiki/comparisons/`、`wiki/analyses/`），并更新索引和日志

### 3. 检查/维护 (Lint)

当用户说"检查知识库"或"健康检查"时：

1. 检查页面间的矛盾
2. 检查被新资料替代的过时说法
3. 检查孤儿页面（没有入链的页面）
4. 检查缺乏页面的重要概念
5. 检查缺失的交叉引用
6. 提出可以通过网络搜索填补的数据缺口
7. 记录结果到 `wiki/log.md`

## 页面格式约定

- **文件名**：使用有意义的名称，如 `思维导图方法论.md`
- **Wiki 链接**：内部链接使用 `[[wiki/entities/xxx]]` 或 `[[wiki/concepts/xxx]]` 格式
- **标签**：在 YAML frontmatter 中使用 `tags:` 字段
- **图片**：引用本地图片使用 `![[raw/assets/xxx.png]]`
- **交叉引用**：页面底部添加"参见"部分，列出相关页面链接

## 命名规范

- `wiki/entities/` 页面：实体名称（人物用中文全名，如 `李开复.md`）
- `wiki/concepts/` 页面：概念名称（如 `第二大脑方法.md`）
- `wiki/sources/` 页面：`[日期]-[简短标题].md`（如 `2026-06-07-知识库设计模式.md`）
- `raw/` 下文件：保持原始文件名，或使用 `[日期]-[来源]-[标题].md`

## 不变原则

1. **永不修改原始资料**——`raw/` 下的文件只读
2. **永不删除用户文件**——除非明确要求
3. **保持一致**——更新索引和日志是所有操作的固定步骤
4. **增量构建**——每次只处理一个资料，循序渐进
